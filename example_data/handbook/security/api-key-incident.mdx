# APIキー漏洩時の初動

APIキーが外部に露出した疑いがある場合は、利用状況を調べる前に失効と再発行を優先します。古いキーを残したまま調査を始めると、不正利用が継続する可能性があります。

## 初動手順

まず開発者ダッシュボードで該当キーを revoke し、新しいキーを発行します。次に影響を受けるジョブやアプリケーションへ新しいキーを配布します。最後に監査ログと利用量ダッシュボードで、通常と異なる呼び出し元や急増したリクエストがないか確認します。

## 通知

顧客データへのアクセスが疑われる場合は、セキュリティ窓口にインシデントチケットを作成します。公開文書に載せてよいのは一般的な初動だけです。内部の調査観点や検知ルールは `/internal/security/api-key-incident-report.mdx` に分けて管理します。

## OAuth との違い

OAuth の `access_token` は短命で、refresh token による更新を前提にできます。API キーは長期間有効になりやすいため、保管場所とローテーションの管理が重要です。
